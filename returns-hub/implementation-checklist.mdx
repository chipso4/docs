Build your API in this order. Each tier builds on the previous one, so you can test incrementally. All endpoints use the V2 base URL: `/api/btu/v2/`.

## Tier 4A: Returns lifecycle

The core return workflow. In V2, all RMA operations use the [`CoreRma`](/data-types/core-objects#corerma) entity. The 15+ V1 PATCH routes collapse into `POST` (create) and `PUT` (update).

<Accordion title="3 V2 endpoints (replaces 15+ V1 endpoints)" defaultOpen>

| Status | Method | V2 endpoint | API reference | Purpose |
|--------|--------|-------------|---------------|---------|
| ☐ | POST | `/api/btu/v2/rma` | [Create return](/api-reference/returns/create-return) | Create an RMA |
| ☐ | GET | `/api/btu/v2/rma/{rmaNumber}` | [Get return](/api-reference/returns/get-return) | Retrieve RMA details |
| ☐ | PUT | `/api/btu/v2/rma/{rmaNumber}` | [Acknowledge](/api-reference/returns/acknowledge) · [Receive](/api-reference/returns/receive) · [Credit](/api-reference/returns/credit-lines) | All lifecycle operations: acknowledge, receive, credit, cancel, update lines, pricing, reason codes, restocking fees, serial numbers |

**What the single `PUT` replaces:** acknowledge, receive, credit-lines, add-lines, cancel, cancel-lines, price, quantity, reason, restocking-fee, return-location, return-type, serial-number — all expressed by updating the relevant fields on the [`CoreRma`](/data-types/core-objects#corerma) entity.

</Accordion>
