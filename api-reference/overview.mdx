## Base URL

```
https://{your-host}/api/btu/v2
```

Replace `{your-host}` with your API's hostname.

## Authentication

All endpoints require an OAuth2 bearer token. See [Authentication](/getting-started/authentication) for details.

```
Authorization: Bearer {access_token}
```

## Hub badges

Each endpoint page shows which hub(s) use it:

- <span style={{background: '#E1F5EE', color: '#0F6E56', padding: '2px 8px', borderRadius: '4px', fontSize: '12px', fontWeight: 500}}>Returns Hub</span> — Non-defect returns and the shared ERP foundation
- <span style={{background: '#EEEDFE', color: '#534AB7', padding: '2px 8px', borderRadius: '4px', fontSize: '12px', fontWeight: 500}}>Warranty Hub</span> — Replacement orders and vendor returns
- <span style={{background: '#FAECE7', color: '#993C1D', padding: '2px 8px', borderRadius: '4px', fontSize: '12px', fontWeight: 500}}>Claims Hub</span> — OEM-facing claim processing

Endpoints with multiple badges are shared — build once, used by multiple hubs.

## V2 design

The V2 API uses **entity-focused REST endpoints** backed by unified Core types. Key changes from V1:

- **One entity, one endpoint** — V1's 17+ RMA PATCH routes collapse into `POST` (create) + `GET` (read) + `PUT` (update) on [`CoreRma`](/data-types/core-objects#corerma)
- **Unified entities** — Customers, vendors, and corporations merge into [`CoreOrganization`](/data-types/core-objects#coreorganization); ship-to/bill-to/sold-to merge into [`CoreLocation`](/data-types/core-objects#corelocation) with type filters
- **Composition** — [`CoreAccountLocation`](/data-types/core-objects#coreaccountlocation) wraps a location with account context; used on invoices, orders, and RMAs for all address roles

| GET | `/api/btu/v2/rma/{rmaNumber}` | [Get return](/api-reference/returns/get-return) | [CoreRma](/data-types/core-objects#corerma) |
| PUT | `/api/btu/v2/rma/{rmaNumber}` | [Acknowledge](/api-reference/returns/acknowledge) · [Receive](/api-reference/returns/receive) · [Credit](/api-reference/returns/credit-lines) | [CoreRma](/data-types/core-objects#corerma) |

<Info>
V1's 15+ PATCH routes (`/return/acknowledge`, `/return/receive`, `/return/credit-lines`, `/return/add-lines`, etc.) all collapse into a single `PUT` on [`CoreRma`](/data-types/core-objects#corerma). The action is expressed by the fields being updated — status changes, line item additions, credit amounts — not by separate routes.
</Info>

## Response conventions

- All responses are `application/json`
- Dates use ISO 8601 format (`2024-01-15` or `2024-01-15T14:30:00Z`)
- All errors follow the [standard error format](/getting-started/error-handling)
- List endpoints support [pagination](/data-types/core-objects#paginatedresponse)
- Every entity supports [`extendedAttributes`](/data-types/core-objects#dataattribute) for custom fields
- All V2 entities are documented in the [Core objects reference](/data-types/core-objects)
