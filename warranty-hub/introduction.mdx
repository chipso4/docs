## What this hub does

The Warranty Hub handles two workflows that sit on top of the Returns Hub foundation:

1. **Replacement orders** — When a defective part is identified, Continuum creates a sales order in your ERP for the replacement. The part is typically picked and handed to the customer at the counter.

2. **Vendor returns** — The defective part needs to go back to the OEM/manufacturer (or be scrapped in field). Continuum creates a vendor return in your ERP to manage that reverse logistics flow.

## The full warranty event

A warranty event spans all three hubs. Here is every API call in order, grouped by phase.

**Phase 0 — Invoice caching** *(background, Returns Hub → ERP)*

| | Method | Endpoint | Returns |
|-|--------|----------|---------|
| 0a | GET | [`/invoices`](/api-reference/invoices/search-invoices) | [CoreInvoice[]](/data-types/core-objects#coreinvoice) |
| 0b | GET | [`/invoice/{invoiceNumber}`](/api-reference/invoices/invoice-details) | [CoreInvoice](/data-types/core-objects#coreinvoice) + lines |

**Phase 1 — Identify the defect** *(Claims Hub → OEM)*

| | Method | Endpoint | Returns | Notes |
|-|--------|----------|---------|-------|
| 1a | GET | [`/validate-unit`](/api-reference/unit-data/validate-unit) | Model, isValid | *Only if brand supports serial lookup* |
| 1b | GET | [`/bill-of-materials`](/api-reference/parts/bill-of-materials) | Parts list + images | Customer selects failed part |
| 1c | GET | [`/products/replacements`](/api-reference/parts/replacements) | Replacement options | Handles superseded parts |
| 1d | GET | [`/products/reason-codes`](/api-reference/parts/warranty-codes) | Failure reason codes | Part-specific |

**Phase 2 — Replace the part** *(Warranty Hub → ERP)*

| | Method | Endpoint | Returns |
|-|--------|----------|---------|
| 2a | GET | [`/parts?checkAvailability=true`](/api-reference/products/availability) | [CorePart](/data-types/core-objects#corepart) + stock |
| 2b | GET | [`/parts?CustomerId={id}`](/api-reference/products/pricing-customer) | [CorePart](/data-types/core-objects#corepart) + price |
| 2c | POST | [`/order`](/api-reference/orders/create-order) | [CoreOrder](/data-types/core-objects#coreorder) |
| 2d | GET | [`/orders?OrderNumbers={id}`](/api-reference/orders/order-status) | [CoreOrder](/data-types/core-objects#coreorder) + invoice status |
| 2e | GET | [`/invoices?OrderNumber={id}`](/api-reference/orders/order-invoices) | [CoreInvoice](/data-types/core-objects#coreinvoice) |

> ⏸ **Gate:** Replacement order must be invoiced before Phase 3.

**Phase 3 — File the warranty claim** *(Claims Hub → OEM)*

| | Method | Endpoint | Purpose |
|-|--------|----------|---------|
| 3a | POST | `/claim/validate` | Pre-validate (dry run) |
| 3b | POST | `/claim` | Submit the claim |
| 3c | POST | `/claim/upload-attachment` | Upload supporting docs |
| 3d | GET | `/claim/status` | Poll for approval |
| 3e | PATCH | `/claim` | Update if corrections needed |

> ⏸ **Gate:** Claim must be approved before Phase 4.

See [Claim flow](/warranty-claims-hub/claim-flow) for full endpoint specs.

**Phase 4 — Handle the defective part** *(Warranty Hub → ERP)*

| | Method | Endpoint | Returns |
|-|--------|----------|---------|
| 4a | POST | [`/vendor-return`](/api-reference/vendor-returns/create-vendor-return) | Vendor return created |
| 4b | PATCH | [`/vendor-return/approve`](/api-reference/vendor-returns/approve) | Approved |
| 4c | PATCH | [`/vendor-return/picked`](/api-reference/vendor-returns/picked) | Picked |
| 4d | PATCH | [`/vendor-return/ship`](/api-reference/vendor-returns/ship) | Shipped |

*Or scrap-in-field — no vendor return created.*

**Phase 5 — Credit the customer** *(Returns Hub → ERP)*

| | Method | Endpoint | Purpose |
|-|--------|----------|---------|
| 5a | PUT | [`/rma/{rmaNumber}`](/api-reference/returns/acknowledge) | Finalize RMA (DRAFT → FINAL) |
| 5b | PUT | [`/rma/{rmaNumber}`](/api-reference/returns/receive) | Receive defective part |
| 5c | PUT | [`/rma/{rmaNumber}`](/api-reference/returns/credit-lines) | Issue credit → offsets replacement order |

<Info>
All endpoints use the V2 base: `/api/btu/v2/`. Phase 3 claim endpoints connect to the OEM, not your ERP — see the [Claims Hub](/warranty-claims-hub/introduction). Phase 5 uses the same [`CoreRma`](/data-types/core-objects#corerma) PUT pattern as the Returns Hub.
</Info>

## Prerequisites

<Warning>
The Warranty Hub shares its entire foundation with the Returns Hub. You must implement the Returns Hub endpoints (organizations, contacts, locations, parts, invoices) before adding Warranty. See the [Returns Hub](/returns-hub/introduction).
</Warning>
