## Design principles

All data types follow the V2 unified entity model (`primary-data-types.json`):

1. **ERP-agnostic** — Classification fields like `organizationType`, `locationType`, and `contactType` use `string` rather than fixed enums. Your middleware maps your ERP's values to whatever string makes sense.

2. **Nullable by default** — All fields are nullable unless marked as **required**. This accommodates ERP systems that may not populate every field.

3. **Extensible via `extendedAttributes`** — Every entity supports a `DataAttribute[]` array for custom fields. This is how you pass product specs, industry-specific data, rate structures, and ERP-specific codes without schema changes.

4. **Composition over embedding** — Address, contact, location, and account-location are standalone types composed into parent entities. A `CoreInvoice` HAS a `CoreAccountLocation` for its shipTo, which HAS a `CoreLocation`, which HAS a `CoreAddress`. No 15-field flat address blocks.

5. **Unified entities** — V1's separate Customer, Vendor, Supplier, and Corporation models are merged into `CoreOrganization` differentiated by `organizationType`. V1's 17+ RMA PATCH endpoints collapse into a single `CoreRma` entity. One shape, one contract.

## Date formats

- **Date only:** `YYYY-MM-DD` (e.g., `"2024-01-15"`)
- **Date and time:** ISO 8601 with timezone (e.g., `"2024-01-15T14:30:00Z"`)

## Field naming

All field names use **camelCase** (e.g., `serialNumber`, `organizationType`, `invoiceNumber`).
