## The three hubs

Continuum's integration surface is organized into three hubs. Each hub serves a different business process, but they share a common foundation.

| Hub | Product tag | Direction | Purpose |
|-----|-------------|-----------|---------|
| **Returns Hub** | `btuReturns` | Continuum → Your ERP | Non-defect product returns. Full RMA lifecycle. Also provides the shared foundation (customers, products, invoices) used by all hubs. |
| **Warranty Hub** | `btuWarranties` | Continuum → Your ERP | Warranty replacement orders and vendor returns. Orchestrates the full warranty event across all three hubs. |
| **Warranty Claims Hub** | `btuWarrantyVendor` | Continuum → OEM API | OEM-facing claim processing. Serial validation, BOM lookups, claim submission to the manufacturer. |

<Info>
You may use one, two, or all three hubs depending on your needs. Each hub has self-contained documentation. But they can also work together as phases in a single warranty event.
</Info>

## Integration architecture

The Returns and Warranty hubs talk to the **distributor's ERP**. The Claims hub talks to the **OEM/manufacturer's API**. These are separate systems with separate authentication.

## Shared foundation

The Returns Hub provides the shared foundation consumed by the Warranty Hub. These endpoints are built once and reused:

- **Authentication** — OAuth2 client credentials (`POST /oauth2/token`)
- **Health** — Service health check (`GET /health`)
- **Customers** — Lookup, contacts, addresses (ship-to, bill-to, sold-to)
- **Products** — Details, availability, pricing, cross-reference
- **Invoices** — Headers, line items, date-range search
- **Locations** — Warehouse locations, bins
- **Reference codes** — RMA reasons, ship vias, adjustment reasons

Build these for Returns. They carry over to Warranty automatically.

## How the hubs orchestrate in a warranty event

When all three hubs are active, a warranty event flows through five phases:

| Phase | What happens | Hub | System |
|-------|-------------|-----|--------|
| **0** | Bulk invoice caching (background) | Returns | ERP |
| **1** | Brand selection → serial/model validation → BOM → select failed part | Claims | OEM |
| **2** | Check inventory → create replacement order → hand part to customer → wait for invoice | Warranty | ERP |
| **3** | Gather claim details → submit to manufacturer → wait for approval | Claims | OEM |
| **4** | Create vendor return (or scrap in field) | Warranty | ERP |
| **5** | Finalize RMA → issue credit that offsets the replacement order | Returns | ERP |

Two gates control the flow: the replacement order must be **invoiced** before the claim is submitted (Phase 2 → 3), and the claim must be **approved** before the vendor return and credit are processed (Phase 3 → 4/5).

## Security

- **Transport:** All API traffic is over HTTPS (TLS 1.2+)
- **Authentication:** OAuth2 client credentials. Continuum presents a bearer token with every request. ERP and OEM endpoints use separate tokens.
- **Network:** You control network access. You can restrict inbound traffic to Continuum's IP ranges, require VPN, or use any other network-level controls.
- **Data:** Your middleware controls exactly what data is exposed. You can filter, redact, or transform any field before returning it.

## Technology stack

Your middleware can be built with any technology stack — .NET, Java, Node.js, Python, Go — as long as it serves an HTTPS REST API conforming to the endpoint contracts in this documentation. Continuum makes outbound HTTPS calls to your middleware. All communication is standard JSON request/response.
